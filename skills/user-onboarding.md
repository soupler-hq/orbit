# Skill: User Onboarding
> Structured framework for onboarding plans, adoption milestones, and first-success guidance

## ACTIVATION
- When a task asks for onboarding documentation, activation planning, or adoption guidance.
- When launch, ship, or docs work needs a clear first-success path for a user segment.
- When `technical-writer` or `launch-planner` is preparing rollout materials tied to adoption.

## CORE PRINCIPLES
1. **Time to Value First**: Optimize for the fastest path to meaningful user success.
2. **Segment-Aware Guidance**: Distinguish between user roles when their setup or goals differ.
3. **Actionable Milestones**: Every onboarding stage must end with a concrete outcome.
4. **Failure-Aware Design**: Anticipate where users will stall and document mitigations early.
5. **Source-Grounded Messaging**: Keep onboarding promises anchored to real product behavior.

## PATTERNS

### Onboarding Journey Map
- **Entry Context**: Why the user is here and what job they are trying to complete
- **Setup Step**: Required configuration, permissions, or integrations
- **First Success**: The earliest visible value moment
- **Expansion Step**: The next action after initial success
- **Habit Loop**: The recurring behavior that signals real adoption

### Time-to-Value Milestones
1. **Day 0**: Access, setup, and orientation
2. **Day 1**: First successful workflow completed
3. **Week 1**: Repeatable usage pattern established
4. **Week 2+**: Expanded feature discovery or team adoption

### Feature Discovery Checklist
- Separate "must use now" from "discover later"
- List prerequisites and compatibility notes before the user hits them
- Provide concrete examples for the first successful workflow
- Clarify which actions belong to admins, operators, or end users

### Common Failure Patterns
- **Setup paralysis**: Too many choices before the first win
- **Value ambiguity**: Users do not know what outcome to expect
- **Feature overload**: Everything is introduced at once
- **Integration drop-off**: Compatibility and troubleshooting appear too late

## CHECKLISTS

### Onboarding Readiness
- [ ] User segments or roles are identified
- [ ] First-success milestone is explicit
- [ ] Time-to-value steps are sequenced
- [ ] Feature discovery is staged into now vs later
- [ ] Failure patterns and mitigations are documented
- [ ] Success criteria are tied to observable user outcomes

## ANTI-PATTERNS
- **Feature Dumping**: Listing capabilities instead of guiding a first success path.
- **Segment Collapse**: Treating all users as if they have identical needs.
- **Empty Milestones**: Using vague steps like "get familiar" without outcomes.
- **Wishful Guidance**: Documenting workflows the product cannot yet support.
- **Late Caveats**: Hiding prerequisites until after the user fails.

## VERIFICATION WORKFLOW
1. **Logical Consistency**: Confirm the onboarding flow matches actual product capabilities and constraints.
2. **Output Integrity**: Verify milestones and checklists are concrete enough for a fresh user or operator to follow.
3. **Adoption Traceability**: Ensure each recommendation maps to a real success signal, workflow, or artifact.
